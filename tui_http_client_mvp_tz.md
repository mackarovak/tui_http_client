# Техническое задание
## MVP TUI HTTP Client на Go

---

## 1. Общее описание

Нужно разработать терминальное приложение на Go в формате TUI (Text User Interface), предназначенное для отправки и сохранения HTTP-запросов.

Продукт по смыслу должен быть **легковесной, простой и дружественной альтернативой Postman** для базовых сценариев работы с API.

Ключевая особенность: приложением должен суметь воспользоваться **пользователь с минимальным опытом работы с HTTP-запросами**. Это значит, что интерфейс, формулировки, сценарии и поведение по умолчанию должны быть ориентированы не только на разработчиков, но и на:
- тестировщиков,
- аналитиков,
- студентов,
- начинающих backend/frontend-разработчиков,
- пользователей, которым нужно «просто отправить запрос и посмотреть ответ».

---

## 2. Цель MVP

MVP должен позволять пользователю без дополнительного обучения:
1. создать новый HTTP-запрос;
2. выбрать метод;
3. указать URL;
4. при необходимости добавить body, headers и Bearer token;
5. отправить запрос;
6. увидеть понятный ответ;
7. сохранить запрос локально для повторного использования.

Если пользователь без глубокого знания HTTP сможет самостоятельно выполнить эти действия, MVP считается успешным.

---

## 3. Позиционирование продукта

Продукт не должен быть полным аналогом Postman.

MVP должен решать только основные задачи:
- быстрый запуск;
- минимум перегрузки интерфейса;
- keyboard-first сценарий;
- локальное хранение запросов;
- понятное отображение ответа;
- низкий порог входа.

Продукт **не обязан** в первой версии поддерживать сложные корпоративные функции, сложные auth flow и автогенерацию из OpenAPI.

---

## 4. Основные принципы UX

### 4.1 Простота по умолчанию
На первом экране пользователь должен видеть только самые важные сущности:
- HTTP method;
- URL;
- body;
- кнопку/действие отправки;
- область ответа.

Все дополнительные детали (`Headers`, `Params`, `Auth`) должны быть вынесены во вкладки или вторичные секции.

### 4.2 Понятные названия
В интерфейсе нельзя перегружать пользователя узкоспециализированными терминами, если можно использовать более понятные.

Предпочтительные названия:
- `Request body` вместо `Payload`;
- `Authentication` вместо `Authorization strategy`;
- `URL parameters` вместо `Query serialization`;
- `Response` вместо `Execution output`.

### 4.3 Пустые состояния с подсказками
Если в интерфейсе ещё нет созданного запроса, пользователь должен видеть не пустую область, а подсказки:
- как создать запрос;
- какой URL можно ввести для примера;
- какую клавишу нажать для отправки.

### 4.4 Безопасные и полезные значения по умолчанию
- при первом запуске должна быть demo-коллекция;
- если пользователь вводит URL без схемы, приложение может автоматически подставить `https://`;
- если выбран GET, body можно скрывать или дизейблить;
- JSON-ответ должен красиво форматироваться автоматически.

### 4.5 Дружелюбные ошибки
Ошибки должны быть понятными человеку без глубоких технических знаний.

Примеры корректных сообщений:
- `Invalid URL`
- `Could not connect to server`
- `Request timed out`
- `Invalid JSON in request body`

Следует избегать непонятных низкоуровневых ошибок без расшифровки, если их можно преобразовать в дружелюбный текст.

## 6. Объём MVP

### 6.1 Что обязательно входит в MVP
1. Создание нового запроса.
2. Редактирование существующего запроса.
3. Выбор HTTP-метода:
   - GET
   - POST
   - PUT
   - PATCH
   - DELETE
4. Ввод URL.
5. Ввод заголовков.
6. Ввод request body.
7. Поддержка режимов body:
   - none
   - raw text
   - JSON
   - x-www-form-urlencoded
8. Простая аутентификация:
   - None
   - Bearer Token
9. Отправка запроса.
10. Просмотр ответа.
11. Сохранение запроса.
12. Удаление запроса.
13. Дублирование запроса.
14. Локальный список сохранённых запросов.
15. Поиск по сохранённым запросам.
16. Demo collection при первом запуске.
17. Экран помощи / список горячих клавиш.

### 6.2 Что желательно включить в MVP, если успевает команда
1. Pretty-print JSON response.
2. Копирование ответа в буфер обмена.
3. Копирование запроса как `curl`.
4. Показ размера ответа.
5. Показ времени выполнения запроса.
6. Адаптация layout под узкий терминал.

### 6.3 Что не входит в MVP
Следующие функции в первую версию **не входят**:
- OAuth2 flow;
- API key wizard с множеством режимов;
- GraphQL playground;
- WebSocket;
- SSE;
- pre-request scripts;
- post-request scripts;
- тестовые assertions;
- импорт OpenAPI/Swagger;
- синхронизация через облако;
- shared workspaces;
- advanced environments с наследованием;
- cookies manager;
- proxy configuration UI;
- multipart file upload в полном виде, если это сильно усложняет MVP.

## 7. Пользовательские сценарии

### 7.1 Основной сценарий: отправка GET-запроса
1. Пользователь открывает приложение.
2. Видит demo-запрос или создаёт новый.
3. Выбирает метод `GET`.
4. Вводит URL.
5. Нажимает `Send`.
6. Получает:
   - статус;
   - время;
   - размер ответа;
   - body ответа.

### 7.2 Основной сценарий: отправка POST JSON
1. Пользователь создаёт новый запрос.
2. Выбирает `POST`.
3. Вводит URL.
4. Выбирает body mode = `JSON`.
5. Вводит JSON.
6. При необходимости задаёт Bearer token.
7. Нажимает `Send`.
8. Если JSON невалидный, получает понятную ошибку до отправки.
9. Если JSON валиден, запрос уходит и ответ отображается.

### 7.3 Повторное использование
1. Пользователь выбирает сохранённый запрос в списке.
2. Меняет URL / body / token.
3. Отправляет повторно.
4. При необходимости сохраняет как новый запрос через дублирование.

## 8. UI / layout

### 8.1 Общая структура экрана
Интерфейс MVP должен состоять из трёх основных зон:

1. **Левая панель** — список сохранённых запросов.
2. **Центральная панель** — редактор запроса.
3. **Правая или нижняя панель** — отображение ответа.

### 8.2 Адаптивность layout
Для широких терминалов:
- список слева,
- редактор по центру,
- response справа.

Для узких терминалов:
- список слева или сверху,
- response переносится вниз.

Точный порог ширины терминала команда может определить самостоятельно, но layout не должен ломаться на стандартных terminal window size.

### 8.3 Экран списка запросов
Каждый элемент списка должен содержать:
- имя запроса;
- HTTP-метод;
- короткий URL.

Пример отображения:
- `Get users    GET    api.site.com/users`
- `Create post  POST   api.site.com/posts`

Должны поддерживаться:
- перемещение по списку;
- выделение активного элемента;
- поиск/фильтрация;
- создание нового запроса;
- удаление;
- дублирование.

### 8.4 Экран редактора запроса
Обязательные элементы:
- поле выбора метода;
- поле URL;
- вкладки или секции:
  - Body
  - Headers
  - Params
  - Authentication

Для упрощения UX по умолчанию фокус должен попадать на URL.

### 8.5 Экран ответа
Обязательные элементы ответа:
- статус (`200 OK`, `404 Not Found`, и т.д.);
- время выполнения;
- размер ответа;
- вкладки:
  - Body
  - Headers

Если response body — JSON, его нужно автоматически форматировать.

Если response body не JSON, показывать как обычный текст.

Если response body бинарный, показать служебное сообщение вроде:
- `Binary response received: 18342 bytes`

## 9. Работа с запросом

### 9.1 HTTP methods
MVP должен поддерживать:
- GET
- POST
- PUT
- PATCH
- DELETE

### 9.2 URL
Требования:
- пользователь может ввести полный URL;
- если схема не указана, приложение может предлагать или автоматически добавлять `https://`;
- URL должен валидироваться перед отправкой;
- при ошибке URL должно показываться понятное сообщение.

### 9.3 URL parameters
Параметры запроса задаются как список пар ключ-значение.

Требования:
- пользователь может добавлять и удалять строки параметров;
- пустые строки не должны ломать сборку URL;
- итоговый URL должен корректно собираться перед отправкой.

### 9.4 Headers
Заголовки задаются как список пар ключ-значение.

Требования:
- пользователь может добавлять и удалять строки;
- заголовки должны сохраняться вместе с запросом;
- если заголовок дублируется вручную и генерируется автоматически (например через Bearer token), приоритет должен быть чётко определён.

Рекомендуемое поведение для MVP:
- автоматически сгенерированный `Authorization` имеет приоритет над пустым пользовательским значением;
- если пользователь явно ввёл свой `Authorization`, команда может либо предупреждать, либо перезаписывать по правилам auth-режима.

Нужно выбрать один вариант и реализовать последовательно.

### 9.5 Request body
Поддерживаемые режимы:
- `none`
- `raw text`
- `json`
- `form-urlencoded`

#### 9.5.1 none
Тело не отправляется.

#### 9.5.2 raw text
Тело отправляется как обычная строка.

#### 9.5.3 json
- пользователь вводит JSON как текст;
- приложение валидирует JSON перед отправкой;
- при ошибке показывает понятное сообщение;
- по возможности автоматически выставляет `Content-Type: application/json`, если пользователь не указал иное.

#### 9.5.4 x-www-form-urlencoded
- ввод через таблицу key-value;
- тело кодируется в соответствии со стандартом `application/x-www-form-urlencoded`.

### 9.6 Authentication
В MVP обязательно поддержать:
- `None`
- `Bearer Token`
Остальные на свое усмотрение

Требования UX:
- токен не должен требовать ручного ввода в Headers;
- поле токена должно быть максимально понятным;
- желательно поддержать скрытие/показ токена;
- если выбран Bearer mode, но token пустой, пользователь должен видеть предупреждение.

## 10. Работа с ответом

### 10.1 Что показывать
После отправки запроса пользователь должен увидеть:
- HTTP status code;
- status text;
- время выполнения;
- размер ответа;
- response headers;
- response body.

### 10.2 Форматирование JSON
Если `Content-Type` указывает на JSON или тело успешно парсится как JSON, response body должен отображаться в pretty-printed виде.

### 10.3 Большие ответы
Для MVP допустимо простое поведение:
- ограничить отображение слишком больших ответов;
- либо показывать body целиком, если это не ломает производительность.

Минимум, который должен быть обеспечен:
- приложение не должно зависать на типичных ответах;
- для больших ответов должно быть прокручиваемое окно.

### 10.4 Ошибки сети и таймауты
Ошибки должны отображаться отдельно от успешного response body.

Пользователь должен видеть, что запрос не выполнился, и по какой причине.

## 11. Хранение данных

### 11.1 Что сохраняется
Сохраняемый запрос должен содержать:
- уникальный ID;
- имя;
- метод;
- URL;
- headers;
- params;
- body mode;
- body;
- auth config.

### 11.2 Где сохраняется
Локально на диске, в пользовательской директории приложения.

Например:
- `~/.config/<appname>/...` на Linux/macOS;
- эквивалентная user config directory на Windows.

Конкретную реализацию команда выбирает в соответствии с нормами Go и ОС.

### 11.3 Первый запуск
При первом запуске приложение должно создавать demo collection с 2–3 примерами:
- простой GET;
- POST JSON;
- запрос с заголовками или токеном.

## 12. Горячие клавиши (опционально)

Для MVP должны поддерживаться следующие действия:
- `n` — новый запрос;
- `Ctrl+Enter` — отправить запрос;
- `Ctrl+S` — сохранить;
- `Tab` — следующая секция;
- `Shift+Tab` — предыдущая секция;
- `/` — поиск;
- `d` — дублировать запрос;
- `Del` — удалить запрос;
- `?` — показать помощь;
- `q` — выход.

Если какие-то сочетания неудобны или конфликтуют с компонентами, команда может предложить эквивалент, но логика должна остаться интуитивной.

## 13. Нефункциональные требования

### 13.1 Производительность
- приложение должно запускаться быстро;
- не должно заметно тормозить на обычных запросах и обычных JSON-ответах;
- UI не должен блокироваться во время выполнения запроса.

### 13.2 Сетевое выполнение
HTTP-запрос должен выполняться асинхронно относительно UI.

Требуется:
- индикатор загрузки / spinner во время запроса;
- возможность отмены длинного запроса, если команда успевает включить это в MVP;
- использование `context.WithTimeout`.

### 13.3 Надёжность
- приложение не должно падать из-за невалидного JSON;
- приложение не должно падать из-за невалидного URL;
- приложение не должно падать из-за пустых полей;
- ошибки файловой системы и сети должны быть обработаны.

## 15. Модели данных (опционально)

Примерная структура, допускаются изменения без потери смысла.

```go
type Header struct {
    Key   string `json:"key" yaml:"key"`
    Value string `json:"value" yaml:"value"`
}

type Param struct {
    Key   string `json:"key" yaml:"key"`
    Value string `json:"value" yaml:"value"`
}

type AuthType string

const (
    AuthNone   AuthType = "none"
    AuthBearer AuthType = "bearer"
)

type AuthConfig struct {
    Type  AuthType `json:"type" yaml:"type"`

type SavedRequest struct {
    ID       string     `json:"id" yaml:"id"`
    Name     string     `json:"name" yaml:"name"`

    Headers  []Header   `json:"headers" yaml:"headers"`
    BodyMode string     `json:"body_mode" yaml:"body_mode"`
    Body     string     `json:"body" yaml:"body"`
    Auth     AuthConfig `json:"auth" yaml:"auth"`

type ResponseData struct {
    StatusCode int      `json:"status_code"`
    StatusText string   `json:"status_text"`
    DurationMs int64    `json:"duration_ms"`
    SizeBytes  int      `json:"size_bytes"`
    Headers    []Header `json:"headers"`
    Body       string   `json:"body"`
    Error      string   `json:"error"`
}
```

## 16. Поведение при отправке

### 16.1 Общая логика
При нажатии `Send` приложение должно:
1. валидировать данные запроса;
2. собрать итоговый URL;
3. применить auth;
4. применить headers;
5. подготовить body;
6. отправить запрос;
7. измерить время выполнения;
8. получить статус, headers и body;
9. отобразить результат.

### 16.2 Валидация до отправки
Минимально обязательны проверки:
- URL заполнен;
- URL валиден;
- если body mode = JSON, JSON валиден;
- если auth mode = Bearer, token не пустой либо пользователь получает предупреждение.

### 16.3 Таймаут
Для MVP должен быть установлен разумный timeout по умолчанию.

Команда может выбрать значение самостоятельно, например 15–30 секунд.

## 19. Критерии приёмки MVP

MVP считается реализованным, если выполняются все пункты ниже.

### 19.1 Создание и сохранение
- пользователь может создать новый запрос;
- пользователь может сохранить его;
- после перезапуска приложения запрос остаётся доступен.

### 19.2 Отправка
- пользователь может отправить GET-запрос;
- пользователь может отправить POST-запрос с JSON body;
- пользователь может отправить запрос с Bearer token.

### 19.3 Ответ
- после запроса отображаются status, time, size и response body;
- JSON-ответ форматируется читаемо;
- response headers можно просмотреть.

### 19.4 UX
- у нового пользователя есть demo requests;
- есть help screen с горячими клавишами;
- ошибки URL/JSON понятны без чтения логов.

### 19.5 Стабильность
- приложение не падает на пустом body;
- приложение не падает на невалидном JSON;
- приложение не падает на ошибке соединения;
- UI остаётся управляемым во время отправки запроса.

## Краткое резюме

Нужно сделать **понятный, быстрый и визуально аккуратный TUI HTTP-клиент на Go**, в котором пользователь без глубоких знаний HTTP сможет:
- создать запрос,
- указать URL,
- добавить JSON body,
- вставить Bearer token,
- отправить запрос,
- увидеть красивый ответ,
- сохранить этот запрос локально.

Главный приоритет MVP — **простота и дружелюбность**.
